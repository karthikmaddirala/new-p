Browsing History RAG

Ask questions about your own browsing history in plain English and get accurate, grounded answers — powered by an LLM-routed hybrid-retrieval pipeline and a local LLM. Nothing leaves your machine.

A local RAG system that continuously syncs your Chrome history, retrieves the most relevant pages with hybrid search (dense vectors + BM25 + Reciprocal Rank Fusion + cross-encoder reranking), and answers with a local Ollama LLM — streamed token-by-token into a React chat UI.

What it does

Type a question like "GitHub repos I visited recently" or "what did I search on Google last week?" and the system:

  1. Routes the query with the LLM — it decides whether to search your history or answer from the conversation so far (a.k.a. adaptive / agentic RAG).
  2. Extracts time — phrases like "last week" become a visit_time filter.
  3. Retrieves with dense (ChromaDB) + sparse (BM25) search in parallel, fused with RRF and deduped by URL.
  4. Reranks the candidates with a cross-encoder and keeps the top 15.
  5. Generates a natural-language answer with Ollama, streamed to the UI as NDJSON.

The frontend is a dark, animated chat interface: a hero prompt with suggested queries, streaming markdown answers, and a live connection indicator.

How retrieval works

Hybrid search beats either method alone:

  • Dense (vector) search captures meaning — "ML tutorials" matches "machine learning courses" — but misses exact/rare keywords.
  • BM25 nails exact keyword and rare-term matches — RecursiveCharacterTextSplitter, qwen2.5:7b — but has no semantic understanding.
  • RRF fuses the two ranked lists using rank position only (1 / (k + rank)), so it needs no score normalization or tuning.
  • A cross-encoder then reranks the fused candidates for a final ordering; the top 15 go to the LLM (no score threshold — the LLM decides relevance).

Time-aware queries ("yesterday", "last week") take a dedicated path: the window is fetched first, then dense + BM25 rank within that window, so every result is guaranteed to be in range. See docs/ARCHITECTURE.md and DECISIONS.md for the full rationale.

Data flow: ingestion

POST /ingest (or sync.py / load_chrome_history.py) → ingestion.ingest(entries):

  1. Filter — skip browser internals (chrome://, about:), redirect/shortener domains, login/error/generic-title pages.
  2. Extract search queries — for known engines (Google, Bing, DuckDuckGo, Yahoo, YouTube, Amazon, Reddit) pull the query param and store "Searched YouTube for: pytorch tutorial". Trivial self-searches are dropped.
  3. Clean titles — strip site-name suffixes (— Site Name), hashtags, emojis, collapse whitespace.
  4. Build the doc string — "{clean title} — {domain}{path}", the unit that gets embedded.
  5. Embed + upsert — batch-embed (batch size 1000) and upsert into ChromaDB with metadata {url, title, domain, visit_time, model_version}. The document id is "{url}#{visit_time}".
  6. Rebuild BM25 — the in-memory index is rebuilt from the full collection.

Pages are stored whole, never chunked — a title/search query is already an atomic unit (see DECISIONS §2).

Data flow: retrieval

retrieval.retrieve(query, k):

  1. Parse time — time_extractor.extract strips temporal phrases and returns days_ago; if present it becomes a visit_time >= now - (days+1)*86400 filter (half-day buffer so "yesterday" covers the full day).
  2. Two paths: - Time filter present: fetch all docs in the window, then run dense (ChromaDB, filtered) and BM25 (filtered) within that set — so nothing in the window is skipped just because it doesn't match the query words. - No filter: dense search for k*4 candidates + BM25 across all docs.
  3. RRF fusion — merge both ranked lists with score += 1 / (60 + rank); scale-invariant, no tuning.
  4. Dedupe by URL — keep the highest-ranked hit per URL.
  5. Return the top k.

Data flow: /chat (agentic + streaming)

POST /chat returns an application/x-ndjson stream. Orchestration in chat_route._stream:

1. yield {"status": "thinking"}            ← opens the HTTP response immediately

2. Router LLM call (temperature=0)         ← decides SEARCH vs ANSWER
      ├─ "ANSWER"          → skip retrieval, answer from conversation
      └─ "SEARCH: <query>" → retrieval.retrieve(query, k=20)
                             → reranker.rerank(...)  → top 15
                             → build context (title + date per hit)

3. Answer LLM call (temperature=0.7, stream)
      for token in llm.stream_chat(...):
          yield {"token": token}

4. yield {"hits": [...], "done": true}     ← final line carries the sources

Two separate LLM calls (router + answerer) keep the answerer's context complete before it generates a token, and prevent follow-up questions like "what were the exact dates?" from wrongly triggering a new search (see DECISIONS §6, §10). The {"status": "thinking"} heartbeat prevents client/proxy timeouts during the router call (DECISIONS §9).

The React frontend reads the stream with a ReadableStream reader, splits on newlines, parses each JSON line, and appends token values live. Conversation history is held client-side and sent with each request — the backend is stateless (DECISIONS §11).

Index lifecycle & the model-version guard

  • BM25 is built in-memory during main.py's lifespan startup and rebuilt after every ingest(). If ChromaDB is empty, _bm25 = None and BM25 queries return [] (safe no-op).
  • Model-version guard: on startup main.py compares the model_version stored in ChromaDB metadata against MODEL_VERSION in config and warns on a mismatch (queries and stored vectors would be from different models). Fix: delete the Chroma data dir and re-run load_chrome_history.py --full. Keep EMBEDDING_MODEL and MODEL_VERSION in sync.

Configuration

All config is env-driven (config.py, loaded from .env). Key variables: EMBEDDING_MODEL / MODEL_VERSION (change together), CHROMA_PATH, TOP_K, OLLAMA_MODEL / OLLAMA_BASE_URL, ENV, LOG_LEVEL. Full reference in DEPLOYMENT.md.

Tests isolate their DB by setting CHROMA_PATH=./chroma_test in tests/conftest.py before any app module imports, so the test DB never touches dev data.

POST /chat

Runs the agentic RAG pipeline and streams the response as newline-delimited JSON (application/x-ndjson). Not a single JSON body — read it as a stream.

Request

{
  "query": "GitHub repos I visited recently",
  "history": [
    { "role": "user", "content": "what did I search last week?" },
    { "role": "assistant", "content": "You searched for ..." }
  ]
}
Field Type Required Notes
query string yes The user's question.
history array of {role, content} no Prior turns. role is "user" or "assistant". Sent by the client each turn (stateless backend).

Response stream

Each line is a standalone JSON object:

{"status": "thinking"}
{"token": "Based"}
{"token": " on"}
{"token": " your"}
...
{"hits": [ ... ], "done": true}
Line Meaning
{"status": "thinking"} Sent immediately to open the connection while the router LLM call runs.
{"token": "..."} One chunk of the streamed answer. Concatenate in order.
{"hits": [...], "done": true} Final line. hits are the retrieved sources (empty if the router chose ANSWER or nothing matched).

Each hit:

{
  "id": "https://github.com/karthikmaddirala#1783250000",
  "score": 0.0312,
  "document": "karthikmaddirala — github.com/karthikmaddirala",
  "distance": 0.41,
  "metadata": {
    "url": "https://github.com/karthikmaddirala",
    "title": "karthikmaddirala · GitHub",
    "domain": "github.com",
    "visit_time": 1783250000,
    "model_version": "mpnet-v1"
  },
  "rerank_score": 4.72
}

Consuming the stream

curl (raw NDJSON):

curl -N -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"query": "YouTube videos I watched recently"}'

JavaScript (as the frontend does):

const res = await fetch('/chat', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ query, history }),
})
const reader = res.body.getReader()
const dec = new TextDecoder()
let buf = ''
while (true) {
  const { done, value } = await reader.read()
  if (done) break
  buf += dec.decode(value, { stream: true })
  const lines = buf.split('\n'); buf = lines.pop()
  for (const ln of lines) {
    if (!ln.trim()) continue
    const obj = JSON.parse(ln)
    if (obj.token !== undefined) appendToAnswer(obj.token)
    if (obj.hits) showSources(obj.hits)
  }
}

Field name: the request field is query, not message.

Kicks off a Chrome-history sync in the background (reads Chrome's SQLite DB via services/sync.py) and returns immediately.

Load your Chrome history

The loader reads Chrome's SQLite history file directly (it copies it first, since Chrome holds a write lock).

python scripts/load_chrome_history.py --full     # all history
python scripts/load_chrome_history.py            # incremental since last run
python scripts/load_chrome_history.py --port 8001

Set your Chrome profile path

The path is hardcoded near the top of the script (and in services/sync.py for background sync):

CHROME_HISTORY_PATH = os.path.expandvars(
    r"%LOCALAPPDATA%\Google\Chrome\User Data\Profile 6\History"
)

Find your profile at chrome://version → Profile Path, and update Profile 6 (and the same constant CHROME_HISTORY in services/sync.py) to match. Tip: the active profile usually has the most recently modified History file.

Background sync

services/sync.py runs automatically inside the backend: an incremental sync ~3s after startup, then every 30 minutes. The last sync timestamp is persisted to scripts/last_sync.json. You can also trigger it via POST /sync.

Chrome extension (optional alternative source)

A Manifest V3 extension (extension/) can push history instead of (or alongside) the SQLite loader.

  1. Open chrome://extensions
  2. Enable Developer Mode
  3. Load unpacked → select the extension/ folder
  4. Browse — background.js backfills the last 30 days on install, then syncs incrementally
  5. Click the extension icon → open the side panel to chat

The backend URL is hardcoded to http://localhost:8000 in extension/background.js — change BACKEND there for other deployments.

Windows auto-start (production)

Run the backend automatically at login via Task Scheduler:

# Run as Administrator
.\scripts\register_startup.ps1

This registers a task that launches scripts/start_server.bat at login (set ENV=production in .env for clean logging).

LLMOps — Evaluation

The llmops/ package is an offline evaluation pipeline for the retrieval and generation quality of the system. It runs against your real chroma_data DB (not the test DB), so populate it first:

python scripts/load_chrome_history.py --full

Metrics are tracked in MLflow (SQLite backend at mlflow.db).

Running an eval

python llmops/run_eval.py                     # retrieval metrics only (domain-based)
python llmops/run_eval.py --judge             # + LLM-as-judge answer quality + faithfulness
python llmops/run_eval.py --ragas             # full RAGAS two-phase eval
python llmops/run_eval.py --label "top_k=20"  # tag this run in MLflow
python llmops/run_eval.py --k 20              # override retrieval depth

mlflow ui                                     # view runs (see run_eval.py header for the port/URI)

Eval cases live in llmops/eval_dataset.json.

Three evaluation modes

1. Retrieval metrics (default) — eval_runner.py

Domain-based, so the eval is stable across days without maintaining exact URLs. For each case it runs retrieval.retrieve(query, k) and compares result domains against the case's expected_domains.

Metric Definition
Domain Recall Of the expected domains, what fraction appeared in top-k?
Domain Precision Of the k results returned, what fraction are from expected domains?
Domain MRR Reciprocal rank of the first result from an expected domain.
Latency (ms) Wall-clock retrieval time (avg + p95).

Cases with an empty expected_domains (results legitimately span many sites) are skipped for recall/precision but still timed.

2. LLM-as-judge — --judge (llm_judge.py)

Generates an answer for each case from the retrieved context, then scores it with the LLM:

  • Answer quality (1–5) — is the response helpful, accurate, well-formed?
  • Faithfulness (0–1) — are all claims grounded in the retrieved context? Unsupported ("hallucinated") claims are listed.

Full per-case outputs (context + answer + score + reasons + hallucinated claims) are logged to MLflow as an answers.md artifact.

3. RAGAS — --ragas (ragas_eval.py)

Standard two-phase RAG evaluation via the RAGAS framework:

Phase 1 — Retrieval
  Context Precision   — are retrieved chunks relevant and well-ranked?
  Context Recall      — does retrieved context cover the ground-truth answer?

Phase 2 — Generation
  Faithfulness        — are answer claims grounded in the context?
  Answer Relevancy    — does the answer address the question asked?

RAGAS uses each case's ground_truth (a description of what a correct answer should contain — not an exact string) for Context Recall.

Eval dataset format

llmops/eval_dataset.json:

{
  "version": "3.0",
  "cases": [
    {
      "id": "youtube-videos",
      "query": "YouTube videos I watched recently",
      "expected_domains": ["youtube.com"],
      "ground_truth": "The user recently watched YouTube videos. The answer should list specific video titles with their visit dates, all sourced from youtube.com.",
      "notes": "Domain check makes sense — YouTube content always comes from youtube.com"
    }
  ]
}
Field Used by Notes
query all The evaluated question.
expected_domains retrieval metrics Domain-level ground truth. Empty = skip recall/precision.
ground_truth RAGAS Context Recall Describes a correct answer; not an exact match.
notes humans Why the case is shaped this way.

Add cases as your history grows to broaden coverage.

MLflow tracking

Each run logs aggregate metrics, per-case metrics, and (with --judge) an answers.md artifact. Use MLflow to compare runs when you tune the pipeline — TOP_K, embedding model, reranking, prompt changes — so improvements are measured, not guessed.

See DECISIONS.md §12 for why the eval is domain-based rather than URL-based.
